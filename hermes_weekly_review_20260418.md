## 二、盈亏比<1的根本原因

### 数学现实

```
期望值 = 胜率 × 平均盈利 - (1-胜率) × 平均亏损
      = 0.684 × 7.11 - 0.316 × 7.92
      = 4.86 - 2.50
      = +$2.36/单（毛利润）
扣佣金后 = +$2.36 - $0.32 = +$2.04/单
```

扣除佣金后，实际盈亏比不是 0.90，而是 **0.77**。系统完全依赖 68% 的高胜率撑着，一旦胜率回落到 55-60%，系统就会亏损。

### 三大根因

**1. 止盈过紧，止损过宽（概率最高）**
- RSRS 均值回归策略特征：回归速度快→止盈易触及→平均盈利小
- 背离时继续背离→止损距离大→平均亏损大
- 标准差过滤太紧，价格刚盈利就被触发平仓

**2. 15分钟扫描频率导致"微利收割"**
- 15分钟级别噪音大，0.1% 强度阈值几乎等于没过滤
- 15分钟 EURUSD 的 0.1% 只有约 1.1 点
- 入场后价格波动空间本身就小

**3. 缺乏动态止盈机制**
- 最佳+18.25 vs 最差-11.48，差距不大
- 说明止盈止损可能是固定的，而非 ATR 动态调整

## 三、USDJPY 亏损的策略层原因

```
USDJPY: 8笔, 胜率25%, 亏损-18.94
NZDUSD: 14笔, 胜率85.7%, 盈利+33.51
```

25% 胜率不是运气问题，是**策略与品种特性不匹配**：

1. **USDJPY 是趋势延续型品种**，RSRS 超买超卖在强势趋势中是假信号
2. **AMA 在 USDJPY 上滞后**，等确认方向时价格已走大部分
3. **因子权重不匹配**：USDJPY 适合"动量+突破"，但 V4.3 给了均值回归 30% 权重

**结论：USDJPY 不是 V4.3 策略应该交易的品种。**

## 四、交易频率过高的原因

```
目标: 5-10单/月 ≈ 1.25-2.5单/周
实际: 38单/周
超标: 15-30倍
```

1. **15分钟扫描 × 24小时 × 5天 = 480次扫描/周**
2. 评分>60 在 15 分钟级别太容易达到
3. 缺乏品种冷却期，AUDUSD 26 笔占 34%

## 五、P0 级改进建议（按优先级）

### P0-1: 立即暂停 USDJPY 交易
- 操作：品种白名单移除 USDJPY
- 效果：每周减少 8 笔亏损，挽回 $18.94+佣金

### P0-2: 降低扫描频率至 1 小时
- 操作：15分钟→60分钟，评分60→70，强度0.1%→0.3%
- 效果：交易降至 8-12 单/周，佣金降至 $2.5-4/周

### P0-3: 引入动态止盈止损
- 操作：止损=ATR(14)×1.5，止盈=ATR(14)×2.0
- 效果：盈亏比从 0.90 提升至 1.3-1.5

### P0-4: 增加品种集中度限制
- 同一品种每周最多 5 笔，平仓后冷却 4 小时
- 效果：AUDUSD 从 26 笔降至 5-8 笔

### P0-5: 引入"市场状态-品种"匹配矩阵
| 市场状态 | 推荐品种 | 禁用品种 |
|----------|----------|----------|
| 趋势市 | USDJPY, USDCAD, GBPUSD | NZDUSD, AUDUSD |
| 震荡市 | NZDUSD, AUDUSD, EURUSD | USDJPY |
| 高波动 | 全部减仓50% | - |

## 七、Hermes 沟通改进建议

### 本次调取过程发现的问题
1. **端口混淆**：8644 是 webhook 平台，API Server 应在 8642 端口（当前未启动）
2. **CLI 非交互模式**：`hermes chat -q "QUERY" -Q --yolo` 是正确的单次查询方式
3. **PowerShell 引号冲突**：Windows PowerShell 和 WSL bash 的引号处理不兼容，需通过文件传递
4. **管道输入不可靠**：cat pipe 方式会导致 Hermes 提前退出，需用 `$(cat file)` 方式

### 建议的标准化流程
1. 数据写入 `/tmp/hermes_input.txt`
2. 创建 shell 脚本调用 `hermes chat -q "$(cat /tmp/hermes_input.txt)" -Q --yolo`
3. 从 WSL 获取输出解析结果
